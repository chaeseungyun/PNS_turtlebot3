# 10 · 하드웨어/센서 물리 구성 (복구된 사실)

> 출처: 사용자 기억 + 원본 노트. 재구성 시 **반드시 반영**해야 할 물리적 제약.
> ⚠️ 표시는 수치/방향을 실측·검증으로 재확인 필요.

## 로봇
- 모델: **TurtleBot3 Waffle Pi** (ROS 2 Humble)
- 로봇 외형 크기(footprint 지름): **약 22 cm** ⚠️ (Nav2 robot_radius/inscribed 계산 기준)
- 매니퓰레이터: 팔은 일시 제거. **단, 팔 하부 마운트가 물리적으로 남아 라이다에 잡힘** → 차폐 필요.

## 라이다 (핵심 제약!)  — ✅ 2026-06-30 실측 확정
- **확정**: `base_link→base_scan` = xyz(-0.064,0,0.122), **rpy(0,0,90°)**. URDF/bringup에서 90° yaw로 보정됨.
- 각도 매핑(scan 프레임): scan 0°=로봇 좌, 90°=후면, 270°=정면.
- 팔 차폐 실측: `/scan` 68-108°(41빔) <0.25m = 매니퓰레이터 하부. 마스크 64-109°가 덮음 → `/scan_filtered`
  에서 제거 확인. **scan_arm_mask.py 마스크 대역 그대로 유효**(변경 불필요).
- LDS 라이다가 **물리적으로 90° 회전 장착**되어 있음 → URDF base_scan yaw=90°로 보정됨(위 확정).
- **방향 매핑** (라이다 기준):
  - 라이다의 **오른쪽 = 로봇 정면(front)**
  - 라이다의 **왼쪽 = 로봇 후면(rear)**
- 라이다 기준 **왼쪽에 매니퓰레이터 하부**가 있어 상시 장애물로 인식됨
  → **좌측 차폐(masking/keepout) 노드** 제작해 해당 각도 스캔 무시/차단.
- ⚠️ 라이다 90° 보정은 "설정값"으로 했는지 "TF(static transform)"로 했는지 재확인 필요.
  - 노트의 미완 마일스톤 "재매핑(라이다 90° 교정 반영한 새 맵)" → **기존 맵은 교정 전 제작이라 폐기 후 재매핑 예정**.

## 카메라 (라이다 사각 보완)  — TF는 ✅ 2026-06-30 확인
- 로봇 전방 카메라를 켜서 **라이다가 못 보는 장애물**(얇은 책상 다리, **바퀴 달린 의자의 바퀴**)을 인식.
- 카메라 인식 결과를 **costmap의 obstacle source로 반영** → 로봇이 회피. (파이프라인 재작성 필요 → [[50-phase2-procedure]] §4)
- **카메라 TF는 이미 존재**: `base_link→camera_link` xyz(0.073,-0.011,0.084) rpy(0,0,0),
  `camera_link→camera_rgb_frame→camera_rgb_optical_frame` 체인 bringup 발행 중.
  → 기본 Pi 카메라 프레임 쓰면 static TF 추가 불필요. (별도 depth 카메라면 그 프레임 확인 필요)
- ❓ 카메라 종류(RGB-D vs RGB 단독)·드라이버·토픽 미확인 → §4 파이프라인 결정에 필요.
